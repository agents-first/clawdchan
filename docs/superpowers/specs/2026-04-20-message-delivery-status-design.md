# Message Delivery Status — Design

**Status:** Approved, 2026-04-20
**Author:** ayham
**Scope:** `core/envelope` (spec-only), `core/node`, `core/store`, `hosts/claudecode`, `cmd/clawdchan`, `docs/design.md`

## Problem

ClawdChan's current send path is fire-and-forget from the sender's point of view. Once `Node.Send` hands an envelope to the transport, the sender has no visibility into whether the relay received it, whether the peer is online, or whether the peer's node stored it. For an async human-in-the-loop protocol, this is a real gap: knowing a question has actually landed on the peer's machine — even before the human reads it — is the difference between "I should wait" and "I should retry or escalate".

## Goal

Give the sender per-envelope visibility into a three-state delivery ladder: **queued → sent → delivered**. Expose that state to the agent via `clawdchan_inbox` and to the human via `clawdchan threads` / `clawdchan listen`.

## Non-goals

- Read receipts. "Delivered to node" is explicitly different from "the human has seen it". The latter is not in scope.
- Detecting policy denial at the peer. A peer whose policy rejects our envelope will leave the sender stuck at `sent` forever — this is correct leak-resistant behavior and is not a bug to fix.
- Retry policy changes. The existing `drainOutboxLoop` behavior — retry forever, no give-up — is preserved unchanged. There is no `failed` terminal state.
- Relay changes. The reference relay in `internal/relayserver` and the wire in `core/relaywire` are untouched.

## State ladder

| State | Meaning | Source of evidence | Trust level |
|---|---|---|---|
| `queued` | Envelope exists in our local store; no successful relay send has occurred yet. | Our own store — nothing else has happened. | — |
| `sent` | Our WebSocket write to the relay returned without error. | Local observation at `core/transport`. Weak but cheap. | Not cryptographically trustworthy; treats the relay as competent but not malicious. |
| `delivered` | The peer's node verified, stored, and returned a signed `IntentAck` envelope naming our `EnvelopeID`. | A signed envelope from the peer's Ed25519 key. | Cryptographically trustworthy. Survives a hostile relay. |

The two levels of trust are deliberate: `sent` is a cheap local-observation optimism, `delivered` is the load-bearing guarantee.

There is no timeout. Status stays at `sent` indefinitely if an ack never arrives. The agent/human interprets staleness using the raw `sent_at_ms` timestamp.

## Wire format

No new `Intent`, no new `Envelope` field, no new CBOR tag, no new relay Ctl kind. The only change is giving the reserved `ParentID` field a meaning *when* `Intent == IntentAck`.

### `IntentAck` envelope shape

| Field | Value |
|---|---|
| `Version` | `1` |
| `EnvelopeID` | fresh ULID |
| `ThreadID` | same as the acked envelope |
| `ParentID` | **the acked envelope's `EnvelopeID`** |
| `From` | acking node's principal. Role is `RoleAgent` — acks are generated by node-level machinery, not by a human principal. |
| `Intent` | `IntentAck` (already reserved in `core/envelope/envelope.go:73`) |
| `CreatedAtMs` | now |
| `Content` | all fields zero (`Kind: ContentText, Text: ""`) |
| `Signature` | Ed25519 over canonical CBOR, identical signing rules to every other envelope |

### Disambiguation rule

`ParentID`'s meaning is Intent-dependent:

- `Intent == IntentAck` → `ParentID` is the `EnvelopeID` being acknowledged.
- `Intent != IntentAck` → `ParentID` remains reserved for future reply-threading (unused in this design; zero-valued).

This keeps each field with one meaning per Intent and does not preclude a future reply-thread feature from using `ParentID` for non-ack intents.

### Spec documentation

`docs/design.md` gains a short subsection under §Intents explaining:

1. That `IntentAck` carries its reference via `ParentID`.
2. That an ack is emitted for every verified, policy-admitted, persisted inbound envelope whose `Intent != IntentAck`.
3. That the ack rides the same sealed-session, signed-envelope, outbox-retry machinery as any other envelope.

## Receive side (peer auto-acks)

In `core/node/node.go:handleInbound` (currently around line 515), after `AppendEnvelope` succeeds and `publish` / `route` are dispatched, the node emits an `IntentAck` if and only if the inbound envelope's `Intent != IntentAck`.

### Flow

```
handleInbound(frame):
    peer = store.GetPeer(frame.From)          # existing
    pt = session.Open(frame.Data)             # existing
    env = envelope.Unmarshal(pt)              # existing
    envelope.Verify(env)                      # existing

    if env.Intent == IntentAck:               # NEW (early-exit branch)
        handleAck(peer, env)
        return

    # existing: thread lookup / auto-create, policy eval
    decision = policy.Evaluate(env, peer)
    if decision == Deny: return               # existing — denied envelopes are NEVER acked
    if decision == Downgrade: env.Intent = NotifyHuman   # existing

    store.AppendEnvelope(env, delivered=true) # existing
    publish(env)                              # existing
    route(env)                                # existing

    emitAck(peer, env)                        # NEW
```

### `emitAck`

Builds, signs, seals, and sends an `IntentAck` envelope back via the normal `deliver` path. On failure, it is enqueued on the outbox like any other envelope. The ack is *not* stored via `AppendEnvelope` on the emitting side (we don't need a record of acks we sent out — they're ephemeral metadata from our perspective), but the emitting side *does* track the ack through the outbox if delivery is deferred.

### Ack's own delivery is not tracked

Acks do not trigger acks (the `env.Intent != IntentAck` guard is the stop rule). Acks also do not have their own delivery status tracked on the sender side — if an ack is ever genuinely lost, the original sender sees `sent` forever. This is acceptable under-reporting: the original envelope *was* delivered, the sender simply does not know.

### Policy gating falls out of ordering

Today `policy.Deny` short-circuits *before* `AppendEnvelope`. Placing `emitAck` *after* `AppendEnvelope` means:

- Denied envelopes → no store record → no ack → sender's status stays `sent` forever.
- Downgraded envelopes (AskHuman → NotifyHuman) → stored with downgraded intent → acked normally.

This is the correct leak-resistant behavior: "stuck on sent" looks the same whether the peer is offline, denied the envelope, or has lost their node entirely. Senders cannot distinguish denial from absence.

## Send side (tracking status)

### Schema change (`core/store/store.go`)

Three new columns on `envelopes`:

```sql
ALTER TABLE envelopes ADD COLUMN status          INTEGER NOT NULL DEFAULT 0;
ALTER TABLE envelopes ADD COLUMN sent_at_ms      INTEGER;  -- nullable
ALTER TABLE envelopes ADD COLUMN delivered_at_ms INTEGER;  -- nullable
```

`status` is a `DeliveryStatus` uint8 persisted as INTEGER. The type lives in `core/store` (not `core/envelope`) because it is persistence metadata, not wire format:

```go
type DeliveryStatus uint8
const (
    StatusQueued    DeliveryStatus = 0
    StatusSent      DeliveryStatus = 1
    StatusDelivered DeliveryStatus = 2
)
```

These columns are only meaningful for outbound envelopes (`from_node == identity.Node`). For inbound envelopes they remain at defaults (`status=0`, timestamps NULL) and are ignored by all surfaces.

### Migration

`core/store/store.go:Open` runs idempotent `ALTER TABLE ... ADD COLUMN` statements after the existing `CREATE TABLE IF NOT EXISTS` block. A helper detects SQLite's "duplicate column name" error so re-running is a no-op. Existing rows receive the column defaults — historical outbound envelopes read as `queued` after upgrade. We do not backfill historical envelopes as `delivered`; we don't actually know they were.

### New `Store` methods

```go
// MarkEnvelopeSent sets status=Sent and sent_at_ms on the given outbound
// envelope. Idempotent and monotonic — never regresses from a higher status.
MarkEnvelopeSent(ctx context.Context, id envelope.ULID, sentAtMs int64) error

// MarkEnvelopeDelivered sets status=Delivered and delivered_at_ms on the
// given outbound envelope. Idempotent; never regresses.
MarkEnvelopeDelivered(ctx context.Context, id envelope.ULID, deliveredAtMs int64) error

// GetOutboundEnvelope returns the stored envelope plus the peer it was sent
// to, if and only if it exists and was originated by this node. Used to
// validate an incoming IntentAck before applying it.
GetOutboundEnvelope(ctx context.Context, id envelope.ULID) (env envelope.Envelope, peer identity.NodeID, ok bool, err error)
```

Both setters use SQL `UPDATE ... WHERE status < ?` to guarantee monotonicity — an older status update arriving after a newer one is a no-op, not a regression.

### State transitions

| Transition | Site | What writes |
|---|---|---|
| (initial → `queued`) | `sendAs` → `AppendEnvelope` | default 0, timestamps NULL |
| `queued → sent` | `deliver` returns `(true, nil)` | `MarkEnvelopeSent(envID, now)` |
| `sent → delivered` | `handleInbound` receives valid `IntentAck` | `MarkEnvelopeDelivered(ackedID, ackCreatedAtMs)` |

Outbox re-drain does not change status — an envelope that failed once and succeeds later still flows through `deliver` returning `(true, nil)`, which calls `MarkEnvelopeSent`. The idempotence guarantee ensures re-setting `sent` on an envelope that's already `delivered` is a no-op.

### Ack-side handling (`handleAck`)

```go
func handleAck(peer pairing.Peer, ack envelope.Envelope):
    # ack.Intent == IntentAck, signature already verified, session already open
    original, expectedPeer, ok, err := store.GetOutboundEnvelope(ack.ParentID)
    if err != nil || !ok:
        return   # we have no record of this envelope — drop silently (strict default)
    if expectedPeer != peer.NodeID:
        return   # ack came from a different peer than the envelope's recipient — drop
    store.MarkEnvelopeDelivered(ack.ParentID, ack.CreatedAtMs)
    publishStatus(original.ThreadID, ack.ParentID, StatusDelivered, ack.CreatedAtMs)
```

The `expectedPeer != peer.NodeID` check blocks a paired peer from acknowledging envelopes we sent to a *different* paired peer. Without it, peer A (legitimately paired, legitimately has a session with us) could spoof delivery confirmations for envelopes we sent to peer B.

Acks are **not** stored via `AppendEnvelope` on the sender side — they are not conversation turns and should not accumulate in the envelopes table on the sending node. The peer's side stores the ack only transiently while it's in the outbox; once delivered, it's not re-recorded.

### `publishStatus` (optional, v2-deferrable)

A new subscription channel type for status-only updates, delivered via a new `Node` method:

```go
func (n *Node) SubscribeStatus(thread envelope.ThreadID) (<-chan StatusEvent, func())
type StatusEvent struct {
    EnvelopeID  envelope.ULID
    Status      store.DeliveryStatus
    AtMs        int64
}
```

This feeds real-time status updates to `clawdchan listen`. For the v1 cut it is **deferred** — the MCP inbox is pull-based and does not need it; the CLI `listen` command can render whatever status the envelope has at publish time and not update in place.

## Surface changes

### MCP (`hosts/claudecode/tools.go`)

`serializeEnvelope` adds three new fields for envelopes with `direction == "out"`:

```json
{
  "envelope_id": "01HQ...",
  "direction": "out",
  "intent": "say",
  "content": {...},
  "created_ms": 1745000000000,
  "status": "delivered",
  "sent_at_ms": 1745000000100,
  "delivered_at_ms": 1745000001250
}
```

- `status`: `"queued"` | `"sent"` | `"delivered"` (string rendering of `DeliveryStatus`).
- `sent_at_ms`: absent when status is `"queued"`, present otherwise.
- `delivered_at_ms`: present only when status is `"delivered"`.

Inbound (`direction == "in"`) envelopes get none of these fields.

`collectInbox` (tools.go:684) also adds a filter: envelopes with `Intent == IntentAck` are skipped before rendering, so acks never appear as conversation turns in the inbox.

`inboxNotes` gains one conditional line, fired only when at least one visible outbound envelope has `status != "delivered"`:

> "Outbound messages show status (queued/sent/delivered). 'queued' means it hasn't left your machine; 'sent' means the relay has it but the peer hasn't acked; 'delivered' means the peer's node stored and signed for it. Persistent 'sent' across hours likely means the peer is offline."

### CLI (`cmd/clawdchan`)

- **`clawdchan threads`** — appends a status glyph to each row, reflecting the status of the most recent outbound envelope on that thread.
- **`clawdchan listen <thread>`** — renders outbound envelopes with a trailing glyph. v1 does not update in place on status transitions (see §publishStatus deferral).

**Glyphs:** `·` queued, `→` sent, `✓` delivered.
**Legacy-Windows fallback:** `.` / `>` / `v`, selected when running on Windows without a terminal signal that indicates a modern Unicode-capable host (e.g., `WT_SESSION` env var unset, and `TERM`/`ConEmu` detection).

Inbound envelopes get no glyph. Ack envelopes are filtered from `listen` output the same way they are from MCP inbox.

## What explicitly does not change

- **`core/envelope`**: no code change, wire format is unchanged.
- **`core/relaywire` and `internal/relayserver`**: Relay semantics unchanged. The existing `"queued"` Ctl continues to be emitted when a peer is offline; the transport's `Events()` channel continues to go unread in `core/node`. We do not add a `"forwarded"` Ctl.
- **`core/session` and `core/transport`**: Acks ride the existing encrypted framing.
- **`core/policy`**: Deny/Admit semantics stay identical. The fact that acks are emitted *after* `AppendEnvelope` means denied envelopes implicitly stay stuck at `sent` on the sender, which is the intended leak-resistant behavior.
- **Retry policy**: `drainOutboxLoop` is unchanged. Retries remain forever-no-give-up.

## Tests

### `core/store/store_test.go`

- `MarkEnvelopeSent` / `MarkEnvelopeDelivered` idempotence — calling twice has no observable effect.
- Monotonicity — calling `MarkEnvelopeSent` on an envelope that's already `delivered` does not regress.
- `MarkEnvelopeDelivered` on a non-existent envelope ID returns `ErrNotFound` (or similar) without panicking.
- `GetOutboundEnvelope` returns `ok=false` for inbound envelopes (envelopes whose `from_node != identity.Node`).
- Migration — opening a pre-migration DB adds the three columns cleanly and does not error on second open.

### `core/node/node_test.go`

- `TestAckRoundtrip`: two nodes paired through an embedded relay; node A sends, eventually node A's envelope reaches `StatusDelivered`; node B never emits an ack for the inbound ack it received.
- `TestAckDoesNotRoute`: an `IntentAck` envelope reaching `handleInbound` does not call any method on `HumanSurface` or `AgentSurface`.
- `TestAckRejectedFromWrongPeer`: paired peer C signs an `IntentAck` referring to an envelope node A sent to peer B; node A's envelope status remains `sent`.
- `TestAckRejectedForUnknownEnvelope`: peer signs an ack for an `EnvelopeID` node A has no record of; no store mutation occurs; no panic.
- `TestAckStatusSurvivesRestart`: node A sends + gets delivered, node A restarts, status is still `delivered` (persistence verification).

### `cmd/clawdchan/cli_integration_test.go`

- Extend the existing compile-and-drive test: send a message between two nodes, poll `threads` until the status glyph flips to `✓`. This verifies the end-to-end wire through store → node → ack → store update → CLI rendering.

Existing tests (envelope signing/verify, inbox shape for inbound, policy eval) remain unchanged and must continue to pass.

## Rollout

Single-PR change. Non-breaking:

- Wire format unchanged — old and new nodes are intercompatible at the protocol level.
- New nodes talking to old nodes: the new node will auto-emit `IntentAck` envelopes. Old nodes receive them, hit their own `handleInbound`, fail to match them to anything (they have no `IntentAck` handling), and by the existing flow the ack gets `AppendEnvelope`d and `route`d. Since `route` has no case for `IntentAck` it falls through to `agent.OnMessage`, which in the Claude Code host is a no-op (per CLAUDE.md). On other hosts this might surface as an empty envelope in a thread — acceptable for transition, and the remedy is upgrading the peer.
- Old nodes talking to new nodes: the old node never emits `IntentAck` envelopes, so the new node's outbound envelopes stay stuck at `sent` forever when talking to old peers. The user sees `→` (no ✓). This is graceful degradation: the new node's local view is accurate ("I have no proof of delivery"), there is no incorrect information. Once the peer upgrades, subsequent envelopes show `✓` normally; historical envelopes stay at `sent` because there's no backfill mechanism.

No feature flag needed. No data migration beyond the idempotent `ALTER TABLE`.
